## 1. Kubernetes 동작 개념 재확인

### Reconciliation Loop (시나리오 1, 2)

실험에서 Pod 삭제 후 8~10초 만에 READY 3/3이 회복됐다. 이것이 Deployment Controller의 **reconciliation loop** 동작이다.

> Reconciliation Loop: Kubernetes가 "원하는 상태(desired state)"와 "현재 상태(current state)"를 지속적으로 비교하고, 차이가 있으면 자동으로 맞추는 반복 동작.
> "누군가 명령을 내리는 것"이 아니라 **상태 차이를 감지하고 자동으로 맞추는 구조**다.

```
etcd(클러스터 상태 저장소)에 저장된 desired state: replicas=3
실제 상태: replicas=2 (Pod 1개 삭제됨)
         ↓
controller-manager(상태 조정 담당 컴포넌트)가 차이 감지
  → kube-apiserver를 통해 etcd의 desired state를 읽고
    현재 실행 중인 Pod 수와 지속적으로 비교한다
         ↓
ReplicaSet Controller에 새 Pod 생성 지시
         ↓
scheduler가 새 Pod를 실행할 노드 결정
         ↓
해당 노드의 kubelet이 컨테이너 실행
         ↓
Readiness Probe 통과 → kube-proxy가 Endpoint 목록 갱신 → 트래픽 합류
```

### RollingUpdate 보호 메커니즘 (시나리오 3)

`maxUnavailable: 1` 실제 동작:

```
기존 ReplicaSet(v1): Pod 3개 실행 중
새 ReplicaSet(v2): nginx:invalid-tag로 배포 시도

1. v2 Pod 1개 생성 시도 → ImagePullBackOff (이미지 없음)
2. maxUnavailable: 1 허용 범위 내에서 v1 Pod 1개 종료
   → 가용 Pod: v1 2개 + v2 0개 = 2개

3. v2가 Ready 상태가 되지 않으므로 v1 추가 종료 중단
   → v1 2개가 계속 트래픽 처리

결론: 잘못된 배포가 전체 장애로 번지지 않음
     단, 자동 rollback은 없음 — 사람이 감지하고 kubectl rollout undo 실행 필요
```

기존 Pod가 3개가 아닌 2개 유지된 것은 `maxUnavailable: 1`의 정상 동작이다.

### Readiness vs Liveness 실제 차이 (시나리오 4, 5)

두 Probe는 역할이 다르지만, Liveness 실패 시 재시작 구간에서 자연스럽게 협력한다.

|                 | 시나리오 4 (Readiness) | 시나리오 5 (Liveness)  |
| --------------- | ---------------------- | ---------------------- |
| 컨테이너 재시작 | ❌ 없음                | ✅ 1회                 |
| Endpoint 제외   | ✅ 9초 내              | ✅ 재시작 중 자동 제외 |
| 자동 복구       | ❌ 수동 필요           | ✅ postStart hook      |
| HTTP 영향       | 없음                   | 없음                   |

**Liveness 실패 시 두 Probe의 협력 흐름:**

```
/livez 파일 삭제
         ↓
kubelet이 Liveness Probe 실패 감지 (5초 주기 × 3회 = 약 15초)
         ↓
kubelet이 컨테이너 재시작 결정
         ↓
컨테이너 재시작 직전 — 컨테이너가 잠시 중단됨
         ↓
Readiness Probe도 실패 (컨테이너가 응답 불가 상태)
         ↓
kube-proxy가 해당 Pod를 Endpoint에서 제외 → 트래픽 차단
         ↓
컨테이너 재시작 완료 → postStart hook 실행 → /livez, /readyz 재생성
         ↓
Readiness Probe 통과 → Endpoint 재합류 → 트래픽 재개
```

Liveness와 Readiness는 독립적으로 설계됐지만, 재시작 구간에서 Readiness가 자동으로 트래픽을 차단해주는 협력 동작이 발생한다.

## 2. 현업 상황과 시나리오 매칭

### 시나리오 1 — Pod 1개 삭제

```
현업 상황:
- 노드 리소스 부족으로 kubelet이 Pod evict (강제 퇴거)
- OOM Killer가 메모리 초과 컨테이너 프로세스 종료
- 노드 OS 패치 후 재부팅 (drain 없이)
- 스팟/프리엠티블 인스턴스 회수
  (클라우드에서 저렴한 대신 언제든 회수될 수 있는 인스턴스 유형)
```

검증된 것: 단일 Pod 장애는 사용자 영향 없이 자동 복구됨

---

### 시나리오 2 — Pod 2개 동시 삭제

```
현업 상황:
- 노드 1대 장애 (해당 노드의 Pod 전체 종료)
- kubectl drain 실행 중 다수 Pod 동시 종료
  (drain: 노드를 점검하기 위해 해당 노드의 Pod를 모두 다른 노드로 이동시키는 명령)
- 배포 파이프라인 버그로 다수 Pod 강제 삭제
```

검증된 것: capacity 33% 구간이 실제로 발생함

- 트래픽이 많은 서비스라면 이 구간에 latency 급증 또는 timeout 가능
- PDB(PodDisruptionBudget) 없이 노드 drain 하면 이 상황이 그대로 발생

---

### 시나리오 3 — 잘못된 이미지 배포

```
현업 상황:
- CI/CD 파이프라인에서 잘못된 태그로 배포 (오타, 삭제된 태그)
- Private registry 인증 만료로 ImagePullBackOff
- 이미지 빌드 실패 후 존재하지 않는 태그 배포
```

검증된 것: RollingUpdate가 전체 장애를 막아줌

- 단, 자동 rollback은 없음 → 사람이 감지하고 rollback 실행해야 함
- 현업에서는 배포 후 일정 시간 내 Readiness 미달 시 자동 rollback하는 도구를 사용한다.
  - 예: Argo Rollouts, Flagger (Progressive Delivery 도구 — 배포를 단계적으로 진행하면서 문제 발생 시 자동으로 이전 버전으로 되돌리는 기능을 제공)

---

### 시나리오 4 — Readiness 실패

```
현업 상황:
- 앱 기동 중 DB 연결 실패로 /health 엔드포인트 503 반환
- 외부 의존 서비스 장애로 앱이 준비 불가 상태
- 설정 파일 누락으로 앱이 초기화 실패
- 배포 직후 warm-up 시간 동안 트래픽 차단 필요
  (warm-up: 앱이 캐시를 채우거나 연결을 초기화하는 준비 시간)
```

검증된 것: Readiness 실패 Pod는 트래픽에서 격리됨

- 현업에서 가장 중요한 Probe. 잘못 설정하면 준비되지 않은 Pod에 트래픽이 계속 유입됨

---

### 시나리오 5 — Liveness 실패

```
현업 상황:
- 앱이 deadlock 상태에 빠져 응답은 없지만 프로세스는 살아있음
  (deadlock: 두 작업이 서로를 기다리며 영원히 멈춰있는 상태)
- 메모리 누수로 앱이 응답 불가 상태
- goroutine/thread leak으로 앱이 hang
- 외부 연결 고갈로 앱이 frozen 상태
```

검증된 것: kubelet이 자동으로 컨테이너를 재시작함

- `failureThreshold`를 너무 낮게 설정하면 일시적 지연에도 불필요한 재시작 발생
- `initialDelaySeconds`를 앱 기동 시간보다 짧게 설정하면 기동 중 재시작 루프(CrashLoopBackOff) 발생
  - CrashLoopBackOff: 컨테이너가 계속 실패하고 재시작을 반복하는 상태

## Steady State 기준

> Steady State(정상 상태): 실험 전후에 시스템이 정상적으로 동작하고 있음을 확인하는 기준값.
> 이 기준을 통과해야 실험을 시작하고, 실험 후에도 이 상태로 돌아와야 복구 완료로 판단한다.

## 시나리오 1 — Pod 1개 삭제

### 가설
Pod 1개 삭제 시 ReplicaSet이 30초 이내에 새 Pod를 생성하고 READY 3/3으로 회복한다.

### 타임라인

| 시각 | 이벤트 |
|------|--------|
| 22:07:26 | T0 Steady State 확인 완료 |
| 22:07:26 | T1 장애 주입 — web-5c987cf954-68dmr 삭제 |
| 22:07:27 | T2 장애 감지 — Pod Terminating, 새 Pod Pending |
| 22:07:28 | T3 복구 시작 — ContainerCreating (컨테이너 이미지 다운로드 및 실행 중) |
| 22:07:34 | T4 READY 3/3 회복 |

### 관찰 결과

- Recovery Time: **8초**
- HTTP 실패 횟수: 0회
- Endpoint 변화: 3 → 2 → 3
  - Endpoint: Service가 트래픽을 보내는 실제 Pod IP 목록. Pod가 삭제되면 즉시 제외된다.
- Pod 상태 전환: `Terminating → Pending → ContainerCreating → Running → 1/1`

### 결론

- 가설 검증: **통과** (SLO 30초 이내)
- ReplicaSet이 Pod 삭제를 즉시 감지하고 새 Pod를 생성했다.
- 새 Pod가 Readiness Probe(준비 상태 검사)를 통과한 후에만 Endpoint에 합류하여 트래픽을 받기 시작했다. 덕분에 아직 준비되지 않은 Pod에 트래픽이 유입되는 상황이 방지됐다.

## 시나리오 2 — Pod 2개 동시 삭제

### 가설
Pod 2개 삭제 시 남은 1개 Pod가 트래픽을 처리하고, 60초 이내에 READY 3/3으로 회복한다.

### 타임라인

| 시각 | 이벤트 |
|------|--------|
| 22:08:06 | T1 장애 주입 — web-5c987cf954-7f9vz, web-5c987cf954-gp9lt 동시 삭제 |
| 22:08:07 | T2 장애 감지 — Endpoint 3 → 1 (10.244.0.11 1개만 남음) |
| 22:08:08 | T3 복구 시작 — 새 Pod 2개 ContainerCreating |
| 22:08:16 | T4 READY 3/3 회복 |

### 관찰 결과

- Recovery Time: **≤ 10초**
- HTTP 실패 횟수: 0회
- Endpoint 변화: 3 → 1 → 2 → 3
- **capacity 33% 구간 발생**: Pod 3개 중 2개가 삭제되어 Endpoint가 1개만 남는 구간이 실제로 관찰됐다. 이 구간에서 모든 트래픽이 Pod 1개에 집중된다. 트래픽이 많은 서비스라면 이 구간에 응답 지연 또는 오류가 발생할 수 있다.

### 결론

- 가설 검증: **통과** (SLO 60초 이내)
- 남은 Pod 1개가 트래픽을 단독 처리하는 구간이 실제로 발생했다.
- 개선 방향: PodDisruptionBudget(PDB) 검토 필요
  - PDB: "동시에 내려갈 수 있는 Pod 수"를 제한하는 Kubernetes 리소스. `minAvailable: 2` 설정 시 항상 최소 2개의 Pod가 살아있도록 강제된다.

## 시나리오 3 — 잘못된 이미지 배포

### 가설
잘못된 이미지 배포 시 기존 Pod 3개가 유지되고, rollback으로 정상 복구된다.

### 타임라인

| 시각 | 이벤트 |
|------|--------|
| 22:09:10 | T1 장애 주입 — nginx:invalid-tag 배포 |
| 22:09:14 | T2 장애 감지 — ErrImagePull → ImagePullBackOff |
| 22:09:41 | T3 복구 시작 — kubectl rollout undo 실행 |
| 22:09:46 | T4 rollback 완료, READY 3/3 |

> - ErrImagePull: 이미지를 registry에서 가져오는 데 실패한 상태
> - ImagePullBackOff: 이미지 pull 실패가 반복되어 재시도 간격을 늘리며 대기 중인 상태

### 관찰 결과

- Recovery Time: **36초** (rollback 포함)
- HTTP 실패 횟수: 0회
- 기존 Pod 유지 현황: **기존 Pod 2개 유지됨** (가설의 "3개"와 다름 — 아래 설명 참고)
- ImagePullBackOff: 신규 Pod 2개에서 발생
- rollback 후 ContainerStatusUnknown 과도 상태 발생 (정상 — rollback 직후 이전 컨테이너 상태를 아직 확인 중인 일시적 상태)

#### 왜 기존 Pod가 3개가 아닌 2개만 유지됐는가?

`maxUnavailable: 1` 설정의 정상 동작이다. RollingUpdate 흐름을 따라가면:

```
초기 상태: 기존 Pod(v1) 3개

1. 새 Pod(v2, invalid-tag) 1개 생성 시도 → ImagePullBackOff
2. maxUnavailable: 1 허용 범위 내에서 기존 Pod(v1) 1개 종료
   → 현재: v1 2개 + v2 0개(실패) = 가용 2개

3. v2가 Ready 상태가 되지 않으므로 v1 추가 종료 중단
   → v1 2개가 계속 트래픽 처리

결과: 기존 Pod 2개 유지 (3개가 아님)
```

가설에서 "기존 Pod 3개 유지"는 틀렸다. `maxUnavailable: 1`이면 최소 2개가 유지된다.

### 결론

- 가설 검증: **통과** (SLO 120초 이내, HTTP 오류 0)
- `maxUnavailable: 1` 설정 덕분에 잘못된 배포가 전체 장애로 번지지 않았다.
- 단, 자동 rollback은 없다. 사람이 직접 `kubectl rollout undo`를 실행해야 한다.
- 개선 포인트: `maxUnavailable: 0`으로 설정하면 기존 Pod를 전혀 내리지 않고 배포 가능 (단, `maxSurge` 값이 있어야 새 Pod를 먼저 생성할 수 있다)

## 시나리오 4 — Readiness 실패

### 가설
Readiness 실패 시 해당 Pod가 Service endpoint에서 제외되고, HTTP 200이 유지된다.

> Readiness Probe: "이 Pod가 지금 트래픽을 받을 준비가 됐는가?"를 주기적으로 검사하는 장치.
> 실패하면 컨테이너를 재시작하지 않고, Service의 Endpoint 목록에서만 제외한다.

### 타임라인

| 시각 | 이벤트 |
|------|--------|
| 22:10:00 | T1 장애 주입 — web-5c987cf954-jb6df 의 /readyz 파일 삭제 |
| 22:10:09 | T2 장애 감지 — Endpoint 3 → 2, readyReplicas=2 |
| 22:10:54 | T3 복구 시작 — /readyz 파일 재생성 (touch /readyz) |
| 22:10:58 | T4 Endpoint 3 → 3 복구 |

### 관찰 결과

- Endpoint 제외까지: **9초** (Probe 주기 3초 × failureThreshold 2회 + 처리 시간)
- HTTP 실패 횟수: 0회
- RESTARTS: 0 — 컨테이너가 재시작되지 않았다. Readiness 실패는 재시작을 유발하지 않는다.
- Endpoint 변화: 3 → 2 → 3

### 결론

- 가설 검증: **통과**
- Readiness Probe 실패는 컨테이너를 재시작하지 않고 Endpoint에서만 제외한다.
- 나머지 2개 Pod가 트래픽을 정상 처리하여 HTTP 200이 유지됐다.
- **자동 복구 없음**: Readiness 실패는 kubelet이 자동으로 고쳐주지 않는다. 이번 실험에서는 수동으로 `/readyz` 파일을 재생성해서 복구했다. 실제 운영에서는 앱 자체가 스스로 준비 상태를 회복하도록 설계해야 한다.

## 시나리오 5 — Liveness 실패

### 가설
Liveness 실패 시 kubelet이 컨테이너를 재시작하고, 재시작 후 정상 상태로 회복된다.

> Liveness Probe: "이 컨테이너가 지금 살아있는가(정상 동작 중인가)?"를 주기적으로 검사하는 장치.
> 실패하면 kubelet(노드에서 Pod를 관리하는 에이전트)이 컨테이너를 강제로 재시작한다.

### 타임라인

| 시각 | 이벤트 |
|------|--------|
| 22:11:16 | T1 장애 주입 — web-5c987cf954-jb6df 의 /livez 파일 삭제 |
| 22:11:29 | T2 장애 감지 — RESTARTS=1, status=Running ready=0/1 |
| 22:11:29 | T3 복구 시작 — kubelet 컨테이너 재시작 |
| 22:11:32 | T4 ready=1/1 자동 회복 |

### 관찰 결과

- Recovery Time: **16초**
- HTTP 실패 횟수: 0회 (재시작 중 Endpoint에서 자동 제외되어 트래픽 차단)
- RESTARTS: 1회 증가
- **자동 복구**: 컨테이너 재시작 시 `postStart hook`이 `/livez` 파일을 자동으로 재생성했다.
  - postStart hook: 컨테이너가 시작된 직후 자동으로 실행되는 명령어. 이 실험에서는 `touch /readyz /livez`로 설정되어 있어 재시작 후 두 파일이 자동 복구됐다.

### 결론

- 가설 검증: **통과** (SLO 60초 이내)
- kubelet이 `failureThreshold: 3`(3회 연속 실패) 도달 후 컨테이너를 재시작했다.
- 재시작 중 Readiness Probe도 함께 실패하여 Endpoint에서 자동 제외됐다 → HTTP 영향 없음
- postStart hook 설계 덕분에 수동 복구 없이 자동 회복됐다.

## 개선 포인트

1. **시나리오 2**: Pod 2개 삭제 시 Endpoint가 1개로 줄어드는 구간 발생
   - 개선: `PodDisruptionBudget(minAvailable: 2)` 적용
   - PDB는 "동시에 내려갈 수 있는 Pod 수"를 제한하는 Kubernetes 리소스다. `minAvailable: 2`로 설정하면 항상 최소 2개의 Pod가 살아있도록 강제된다.

2. **시나리오 3**: 기존 Pod가 2개까지 줄어드는 구간 발생
   - 개선: `maxUnavailable: 0`으로 설정하면 기존 Pod를 전혀 내리지 않고 배포 가능
   - 단, `maxSurge` 값이 있어야 새 Pod를 먼저 생성할 수 있다. (`maxUnavailable: 0`, `maxSurge: 1` 조합 권장)

3. **시나리오 4**: Readiness 실패는 자동 복구되지 않음
   - 개선: 실제 운영에서는 앱이 스스로 준비 상태를 회복하도록 설계해야 한다. (예: DB 재연결 로직, 헬스체크 엔드포인트 자가 복구)
